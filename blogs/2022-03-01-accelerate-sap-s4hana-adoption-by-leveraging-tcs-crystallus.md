<h2>Deployment of TCS Crystallus in SAP Landscape</h2> 
<p>Below is a sample architecture which depicts the details of how TCS Crystallus on AWS as a reference system is deployed on AWS in a customer landscape, along with non-production and mission-critical production SAP systems.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/77de68daecd823babbb58edb1c8e14d7106e83bb/2022/02/20/TCS-SAP-Crystallus-3.2.png"><img alt="TCS-SAP-Crystallus-3.2" class="aligncenter size-full wp-image-29791" height="741" src="https://d2908q01vomqb2.cloudfront.net/77de68daecd823babbb58edb1c8e14d7106e83bb/2022/02/20/TCS-SAP-Crystallus-3.2.png" width="1253" /></a></p> 
<p style="text-align: center;"><em>Figure 3 – TCS Crystallus on AWS architecture in a customer landscape.</em></p> 
<p>In this design, SAP landscape is split into multiple tiers like Development (DEV), Integration Testing (QA), Production (PROD), Shared Core Services, and Crystallus Golden Template.</p> 
<p>Each tier is assigned with a separate <a href="https://aws.amazon.com/vpc/">Amazon Virtual Private Cloud</a> (VPC) and <a href="https://aws.amazon.com/transit-gateway/">AWS Transit Gateway</a> acts as a hub which connects the VPCs together. This hub-and-spoke model ensures isolation of different parts of infrastructure and allows more precise access controls.</p> 
<p>TCS Crystallus S/4HANA instance is deployed through <a href="https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/AMIs.html">Amazon Machine Images</a> (AMIs). The rest of non-production and production SAP instances are deployed in their respective VPCs with a combination of custom-built AWS CloudFormation templates, Terraform modules, and <a href="https://aws.amazon.com/launchwizard/">AWS Launch Wizard</a>. This infrastructure as code (IaC) process of deploying SAP on AWS cuts down the build timeline from weeks to hours.</p> 
<p>The deployed Crystallus S/4HANA system will serve as a golden template for the rest of the landscape. Fit-to-standard and customer-specific templates are generated here after Fit/Gap workshop, and findings are transported into the development system for further enhancement and unit testing.</p> 
<p>These changes are moved to the quality system, and after successful integration testing they are finally moved to the production landscape. This would now contain the industry best practices and customer-specific customizations generated through Fit/Gap workshop.</p> 
<p>The production landscape is built upon SAP and AWS best practices to match the customer’s system uptime requirement. In the architecture above, the production SAP S/4HANA system is configured with high availability clustering. The SAP application layer’s point of failure being ABAP central services (ASCS) in Availability Zone 1 (AZ1) is configured with lock replication to SAP enqueue replication server (ERS) in AZ2.</p> 
<p>Similarly, the primary SAP HANA database (AZ1) is connected to secondary SAP HANA database (AZ2) via HANA system replication (HSR). SUSE Enterprise Linux (SLES) or Red Hat Enterprise Linux (RHEL) offered SAP-certified clustering packages are configured. This is both application and AWS infrastructure-aware to handle failover of resources from primary AZ1 to secondary AZ2, thereby ensuring maximum uptime of the business critical SAP S/4HANA system.</p> 
<p>In addition to the primary landscape, which is configured with high availability to meet system uptime requirements, for business continuity this landscape can be attached with disaster recovery (DR) designs like warm standby DR, pilot light DR, or hot-standby DR to suit the expected Recovery Time Objective (RTO) and Recovery Point Objective (RPO).</p>
